# AWS 개요

## 정의
Amazon Web Services (AWS)는 Amazon이 제공하는 클라우드 컴퓨팅 플랫폼으로, 전 세계에 분산된 데이터 센터를 통해 인프라, 플랫폼, 소프트웨어를 서비스로 제공합니다.

### 클라우드 컴퓨팅이란?
클라우드 컴퓨팅은 인터넷을 통해 컴퓨팅 리소스(서버, 스토리지, 데이터베이스, 네트워킹, 소프트웨어 등)를 제공하는 방식입니다. 전통적인 온프레미스(자체 데이터 센터) 방식과 달리, 클라우드는 다음과 같은 장점을 제공합니다:

- **온디맨드 제공**: 필요할 때 즉시 리소스를 사용할 수 있음
- **확장성**: 수요에 따라 자동으로 리소스 조정
- **비용 효율성**: 사용한 만큼만 비용 지불 (종량제)
- **글로벌 접근성**: 전 세계 어디서나 접근 가능
- **관리 부담 감소**: 인프라 관리 대신 애플리케이션 개발에 집중

## 클라우드 컴퓨팅 서비스 모델

AWS는 세 가지 주요 서비스 모델을 제공합니다:

### 1. IaaS (Infrastructure as a Service) - 인프라 서비스
**정의**: 가상화된 컴퓨팅 리소스를 인터넷을 통해 제공하는 서비스 모델

**특징**:
- 사용자가 서버, 스토리지, 네트워킹을 직접 관리
- 운영 체제, 미들웨어, 런타임, 데이터, 애플리케이션까지 관리 책임
- 가장 많은 제어권과 유연성 제공

**AWS 서비스 예시**:
- **EC2 (Elastic Compute Cloud)**: 가상 서버
- **VPC (Virtual Private Cloud)**: 가상 네트워크
- **S3 (Simple Storage Service)**: 객체 스토리지
- **EBS (Elastic Block Store)**: 블록 스토리지

**사용 사례**: 
- 기존 온프레미스 애플리케이션을 클라우드로 마이그레이션
- 완전한 제어가 필요한 엔터프라이즈 애플리케이션
- 커스텀 인프라 구성이 필요한 경우

### 2. PaaS (Platform as a Service) - 플랫폼 서비스
**정의**: 애플리케이션 개발 및 배포를 위한 플랫폼을 제공하는 서비스 모델

**특징**:
- 인프라 관리 부담 제거
- 개발 환경, 런타임, 미들웨어는 AWS가 관리
- 개발자는 애플리케이션 코드와 데이터에만 집중

**AWS 서비스 예시**:
- **Elastic Beanstalk**: 애플리케이션 배포 및 관리 자동화
- **Lambda**: 서버리스 컴퓨팅 (서버 관리 불필요)
- **ECS/EKS**: 컨테이너 오케스트레이션 플랫폼

**사용 사례**:
- 빠른 애플리케이션 개발 및 배포
- 개발팀이 인프라 관리에 시간을 쓰고 싶지 않을 때
- 마이크로서비스 아키텍처 구축

### 3. SaaS (Software as a Service) - 소프트웨어 서비스
**정의**: 완전한 소프트웨어 애플리케이션을 인터넷을 통해 제공하는 서비스 모델

**특징**:
- 모든 인프라, 플랫폼, 소프트웨어를 AWS가 관리
- 사용자는 웹 브라우저나 클라이언트 애플리케이션을 통해 접근
- 유지보수, 업데이트, 보안 패치 모두 AWS 책임

**AWS 서비스 예시**:
- **WorkSpaces**: 데스크톱 가상화 서비스
- **Chime**: 화상 회의 및 통신 서비스
- **WorkDocs**: 문서 관리 및 협업 서비스

**사용 사례**:
- 즉시 사용 가능한 소프트웨어가 필요한 경우
- 인프라나 플랫폼 관리가 전혀 필요 없는 경우

## AWS 글로벌 인프라 구조

AWS는 전 세계에 분산된 인프라를 통해 고가용성과 낮은 지연 시간을 제공합니다.

### 리전 (Region)
**정의**: 지리적으로 분리된 독립적인 데이터 센터 그룹

**특징**:
- 각 리전은 완전히 독립적 (다른 리전의 장애에 영향 없음)
- 최소 2개 이상의 가용 영역으로 구성
- 리전 간 데이터 전송 시 비용 발생
- 리전별로 서비스 가용성과 가격이 다를 수 있음

**선택 기준**:
- **규정 준수**: 데이터 거버넌스 요구사항 (예: GDPR)
- **지연 시간**: 사용자와의 지리적 근접성
- **비용**: 리전별 가격 차이
- **서비스 가용성**: 일부 서비스는 특정 리전에서만 제공

**예시**: 
- `ap-northeast-2` (서울 리전)
- `us-east-1` (버지니아 북부)
- `eu-west-1` (아일랜드)

### 가용 영역 (Availability Zone, AZ)
**정의**: 리전 내에 위치한 독립적인 데이터 센터

**특징**:
- 각 AZ는 독립적인 전원, 네트워크, 냉각 시스템 보유
- AZ 간 낮은 지연 시간 (일반적으로 1-2ms)
- 단일 AZ 장애 시 다른 AZ가 자동으로 처리
- 고가용성을 위해 최소 2개 이상의 AZ에 리소스 배포 권장

**작동 원리**:
```
리전 (서울)
├── 가용 영역 A (ap-northeast-2a)
│   └── 독립 데이터 센터
├── 가용 영역 B (ap-northeast-2b)
│   └── 독립 데이터 센터
└── 가용 영역 C (ap-northeast-2c)
    └── 독립 데이터 센터
```

### 엣지 로케이션 (Edge Location)
**정의**: CloudFront와 Route 53을 위한 글로벌 캐시 위치

**특징**:
- 리전보다 훨씬 많은 수의 엣지 로케이션 존재 (전 세계 수백 개)
- 콘텐츠를 사용자와 가까운 위치에 캐시하여 지연 시간 감소
- 정적 콘텐츠 및 동적 콘텐츠 가속화

**사용 사례**:
- 전 세계 사용자에게 빠른 콘텐츠 전송
- 비디오 스트리밍
- 웹사이트 정적 자산 (이미지, CSS, JavaScript) 배포

## AWS 주요 서비스 범주

### 컴퓨팅 서비스
**목적**: 애플리케이션 실행을 위한 컴퓨팅 리소스 제공

- **EC2**: 가상 서버 (온디맨드, 예약형, 스팟 인스턴스)
- **Lambda**: 서버리스 컴퓨팅 (이벤트 기반 실행)
- **ECS/EKS**: 컨테이너 오케스트레이션
- **Fargate**: 서버리스 컨테이너 실행

### 스토리지 서비스
**목적**: 데이터 저장 및 백업

- **S3**: 객체 스토리지 (무제한 확장)
- **EBS**: 블록 스토리지 (EC2 볼륨)
- **EFS**: 파일 스토리지 (공유 파일 시스템)
- **Glacier**: 아카이브 스토리지 (장기 보관)

### 데이터베이스 서비스
**목적**: 구조화된 데이터 저장 및 관리

- **RDS**: 관리형 관계형 데이터베이스 (MySQL, PostgreSQL 등)
- **DynamoDB**: 관리형 NoSQL 데이터베이스
- **Aurora**: 고성능 관계형 데이터베이스
- **ElastiCache**: 인메모리 캐시 (Redis, Memcached)

### 네트워킹 서비스
**목적**: 네트워크 연결 및 트래픽 관리

- **VPC**: 가상 네트워크 (격리된 네트워크 환경)
- **CloudFront**: 콘텐츠 전송 네트워크 (CDN)
- **Route 53**: DNS 서비스
- **API Gateway**: API 관리 및 게이트웨이

### 보안 서비스
**목적**: 리소스 보안 및 액세스 제어

- **IAM**: 사용자 및 권한 관리
- **KMS**: 암호화 키 관리
- **Shield**: DDoS 보호
- **WAF**: 웹 애플리케이션 방화벽

### 관리 및 모니터링 서비스
**목적**: 리소스 관리, 모니터링, 자동화

- **CloudWatch**: 모니터링 및 로깅
- **CloudFormation**: 인프라 코드화
- **Systems Manager**: 운영 관리
- **Config**: 리소스 구성 추적

## 핵심 개념

### 온디맨드 (On-Demand)
**정의**: 필요할 때만 리소스를 사용하고, 사용한 만큼만 비용을 지불하는 방식

**특징**:
- 예측 불가능한 워크로드에 적합
- 약정이나 선불 비용 없음
- 언제든지 시작하고 중지 가능
- 가장 유연하지만 비용이 높을 수 있음

**예시**:
- 개발/테스트 환경
- 트래픽 패턴을 예측하기 어려운 애플리케이션
- 단기 프로젝트

### 확장성 (Scalability)
**정의**: 워크로드 증가에 대응하여 리소스를 늘리는 능력

**확장 유형**:

1. **수평 확장 (Horizontal Scaling / Scale Out)**
   - 리소스의 **수량**을 늘리는 것
   - 예: EC2 인스턴스를 2개에서 10개로 증가
   - 장점: 무중단 확장 가능, 단일 장애 지점 제거
   - 단점: 애플리케이션이 수평 확장을 지원해야 함

2. **수직 확장 (Vertical Scaling / Scale Up)**
   - 리소스의 **사양**을 늘리는 것
   - 예: EC2 인스턴스를 t2.micro에서 m5.large로 업그레이드
   - 장점: 애플리케이션 변경 불필요
   - 단점: 다운타임 발생, 하드웨어 한계 존재

**실제 예시**:
```
수평 확장:
웹 서버 1대 (처리량: 1000 req/s)
  → 웹 서버 5대 (처리량: 5000 req/s)

수직 확장:
웹 서버 1대 (CPU: 1코어, 메모리: 1GB)
  → 웹 서버 1대 (CPU: 4코어, 메모리: 16GB)
```

### 탄력성 (Elasticity)
**정의**: 워크로드 변화에 따라 자동으로 리소스를 조정하는 능력

**확장성과의 차이**:
- **확장성**: 리소스를 늘릴 수 있는 능력 (수동 또는 자동)
- **탄력성**: 자동으로 늘리고 줄이는 능력 (자동화 필수)

**작동 원리**:
```
트래픽 증가
  ↓
CloudWatch 지표 모니터링 (CPU 사용률 > 70%)
  ↓
Auto Scaling 정책 트리거
  ↓
EC2 인스턴스 자동 추가
  ↓
트래픽 감소
  ↓
CloudWatch 지표 모니터링 (CPU 사용률 < 30%)
  ↓
Auto Scaling 정책 트리거
  ↓
EC2 인스턴스 자동 제거
```

**장점**:
- 비용 최적화: 필요할 때만 리소스 사용
- 성능 보장: 트래픽 증가 시 자동 대응
- 운영 부담 감소: 수동 개입 불필요

## AWS의 핵심 가치

### 1. 서비스 선택 기준
워크로드의 요구사항에 맞는 서비스를 선택하는 것이 중요합니다:

- **성능 요구사항**: 높은 성능이 필요한가?
- **확장성 요구사항**: 예측 가능한가, 불가능한가?
- **비용 제약**: 예산이 제한적인가?
- **관리 부담**: 직접 관리할 수 있는가?
- **규정 준수**: 특정 규정을 준수해야 하는가?

### 2. 비용 최적화
사용 패턴에 따라 최적의 구매 옵션을 선택:

- **예측 가능한 워크로드**: 예약형 인스턴스 (최대 72% 할인)
- **유연한 워크로드**: 스팟 인스턴스 (최대 90% 할인)
- **변동적인 워크로드**: 온디맨드 인스턴스

### 3. 고가용성 설계
시스템이 중단 없이 작동하도록 설계:

- **Multi-AZ 배포**: 여러 가용 영역에 리소스 분산
- **Multi-Region 배포**: 여러 리전에 리소스 분산
- **자동 장애 조치**: 장애 발생 시 자동으로 대체 리소스로 전환

### 4. 보안
공동 책임 모델을 이해하고 적절한 보안 조치:

- **AWS 책임**: 클라우드 인프라 보안
- **고객 책임**: 클라우드 내 리소스 보안 (데이터, 애플리케이션, 액세스 제어)
